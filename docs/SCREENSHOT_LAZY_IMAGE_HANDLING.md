# 스크린샷(사이트 미리보기)에서 키비주얼 이미지가 간헐적으로 비는 문제

이 문서는 분석 결과 화면(Overview)의 사이트 미리보기 이미지가 일부 사이트에서 **키비주얼(상단 히어로) 영역 이미지가 로드되기 전에 캡처되어 빈 상태로 보이는 현상**의 원인과, 코드에서 적용한 개선 방향을 정리합니다.

---

## 1. 증상

- 분석 결과 화면의 미리보기 스크린샷에서, 상단 키비주얼 이미지가 간헐적으로:
  - 스켈레톤/플레이스홀더로 남아 있거나
  - 빈 박스처럼 보이거나
  - 낮은 해상도 프리뷰 상태로 보입니다.

---

## 2. 원인(왜 “간헐적”으로 발생하나)

이 프로젝트의 분석은 Puppeteer가 페이지로 이동한 뒤, DOM/axe/Lighthouse 등을 수집하고 스크린샷을 찍습니다. 이때 **스크린샷 캡처 타이밍**과 사이트의 **이미지 로딩 전략**이 어긋나면 키비주얼이 완전히 렌더되기 전에 캡처가 찍힐 수 있습니다.

대표적인 원인:

- **lazy 로딩(IntersectionObserver)**
  - 뷰포트에 들어와도 `src/srcset`가 한 템포 늦게 설정되거나, 스크롤/리사이즈 트리거가 필요한 구현이 있습니다.
- **JS 교체 렌더링**
  - 초기에는 스켈레톤을 렌더하고, 이후 API/스토어/라우트 안정화 후 실제 히어로를 교체하는 SPA 패턴이 있습니다.
- **키비주얼이 `<img>`가 아닐 수 있음**
  - CSS background-image, `<picture>`, `<video>`, canvas 등은 “document.images 완료 여부”로는 완전하게 커버되지 않습니다.
- **네트워크/캐시 변동**
  - CDN 응답 속도, 리다이렉트, 캐시 hit/miss에 따라 로딩 완료 시점이 매번 달라집니다.

즉 “항상 1초 더 기다리면 된다” 같은 단순 문제가 아니라, 사이트별 로딩 방식과 네트워크 상태에 따라 **캡처 시점이 앞서거나 뒤서면서** 간헐적으로 보입니다.

---

## 3. 기존 동작(캡처가 언제 찍히는지)

구현: `lib/services/analyzer.ts`

- `page.goto(..., waitUntil: 'domcontentloaded')` 이후
- `scrollPageForLazyContent(page)`로 lazy 콘텐츠를 유도
- 1차 캡처(폴백용) 후,
- `settlePageDomForArchitecture(page)`로 네트워크 유휴/RAF 등 “정착”을 시도한 다음
- 2차 캡처를 찍어 최종 스크린샷으로 사용합니다. (2차 스냅샷이 실패하면 1차로 폴백)

---

## 4. 개선 방향(이번 변경)

### 핵심 아이디어

“정착(2차) 후 캡처”는 이미 존재하므로, **2차 캡처 직전에 뷰포트 내 이미지 로딩 상태를 짧게 확인하고, 로딩이 덜 됐으면 제한 시간 내에서만 대기**한 뒤 캡처합니다.

### 적용 방식

- 2차 캡처 직전에 `waitForViewportImagesLoaded()`를 호출합니다.
- 이 함수는 브라우저 컨텍스트에서:
  - 뷰포트(화면)에 겹치는 `<img>`들을 찾아
  - `img.complete && img.naturalWidth > 0`가 될 때까지
  - 최대 `VIEWPORT_IMAGE_WAIT_MS`(기본 2500ms) 동안 대기합니다.
- 타임아웃이 나도 분석 전체가 실패하지 않도록 **대기 실패는 무시하고 캡처를 진행**합니다.

효과:

- “히어로가 `<img>` 기반이고, 로딩이 약간 늦는” 사이트에서 빈 키비주얼 캡처 빈도를 줄입니다.
- 대신 **무한 대기/전체 지연**을 피하기 위해 대기는 상한을 둡니다.

한계:

- 키비주얼이 CSS background/video/canvas인 경우에는 이 대기가 직접적인 해결책이 아닐 수 있습니다.
- lazy 로딩 구현이 `img.complete`와 무관하게 별도 렌더링 단계를 거치면, 여전히 간헐적 누락이 있을 수 있습니다.
